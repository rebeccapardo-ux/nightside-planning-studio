## Decisions needed before further fixing

The brief explicitly asks me to surface these.

### A) Focus visible style (pick one)

Option 1 — **Outline-based**:
```css
:focus-visible {
  outline: 2px solid #2C3777;   /* Night */
  outline-offset: 2px;
  border-radius: inherit;
}
```
Pros: simple, browser-native, respects existing border-radius.
Cons: outline doesn't always animate smoothly; can clip outside parent on tightly-nested layouts.

Option 2 — **Box-shadow ring**:
```css
:focus-visible {
  outline: none;
  box-shadow: 0 0 0 3px rgba(44, 55, 119, 0.40);   /* Night at 40% */
}
```
Pros: softer; layers nicely; works on rounded elements without clipping; can be selectively overridden by `box-shadow:none, outline:auto` per element if needed.
Cons: replaces any existing box-shadow unless we compose; requires composing on components that already have shadows.

My recommendation: **Option 2** for consistency with the soft brand aesthetic, applied via a `:focus-visible` rule in `globals.css` and selectively layered on shadowed components.

### B) Brand-color contrast handling

Three colors are failing:
- **Dusk `#BBABF4`** — used in 100+ spots, but mostly as **backgrounds** (lavender panels, accents, modals). Dark text on Dusk: `#130426` on `#BBABF4` = 7.0 : 1 ✅, which is fine. The problem is when Dusk is used as **text** on a cream background (rare, but needs to be verified).
- **Sunset `#DB5835`** — borderline (4.00 : 1). Passes for large text (3:1) which covers headings ≥24px or 18px+bold. Below that, fails.
- **Sunrise `#F29836`** — fails everywhere it carries text.

Per the brief I'm not changing hex values. Three options to apply where these colors carry critical text:

1. **Swap text color to Night / Midnight** when the brand color is purely decorative for the affected element — keeps brand identity in adjacent elements but moves the actual reading text to an accessible value.
2. **Darken the background** behind the brand-color text — e.g., put Sunset text on a darker cream variant.
3. **Use the brand color only for headings ≥24px** (large-text threshold of 3:1 is comfortable for Sunset, still too low for Sunrise/Dusk).

Need a sweep of the 230 occurrences to bucket each into "text role" vs "decorative role" before I can propose specific fixes. Want me to do that sweep next?

### C) `media-has-caption` on voice notes

The audio elements in `VoiceNoteCard`, `VoiceNotePlayback`, `VoiceNoteRecorder` already have an associated **transcript** rendered alongside the audio. The lint rule wants a `<track kind="captions">`. Options:

1. **Disable the rule per-occurrence** with a lint-disable comment + `aria-describedby` pointing to the transcript element. Justification documented inline.
2. **Generate a WebVTT track file** at transcription time and attach it to the audio element. Cleaner, more standards-compliant; more engineering effort.

Option 1 is pragmatic for this audit pass; Option 2 is a roadmap item. Your call.

### D) Runtime audit infrastructure (axe / Lighthouse)

Three viable approaches for the route-level audit:

1. **Manual spot-checks via Chrome DevTools Lighthouse** — fastest to start, free, no infra. You run them yourself per route. I document the findings.
2. **`@axe-core/cli` + Playwright + a session cookie helper** — I write a small script that authenticates as a test user, navigates the route list, and dumps axe JSON to `audits/axe-output/`. Repeatable, but ~half a day of setup.
3. **CI integration** (axe + Lighthouse as a GitHub Action against deployed Vercel preview) — full-fat solution, ~a day of setup, but pays back on every PR.

Recommend (1) for this initial pass, with (2) as a stretch if results are noisy enough to warrant repeated runs.

> **3. Are there any existing accessibility considerations already in the codebase I should know about before adding new patterns?**

Yes:
- `prefers-reduced-motion` is already respected in 7 places (footer back-to-top, both tours, both rankings, legacy map tooltip, scroll helper). Pattern: `window.matchMedia('(prefers-reduced-motion: reduce)').matches` or `@media (prefers-reduced-motion: reduce)` in CSS.

- `data-tour-anchor` attributes have been added to several elements for the tour system — don't conflict with anything.
- No `:focus-visible` styles are defined globally — the codebase has been relying on browser defaults so far.
- The capture documents and activity pages have a `loading` gate that early-returns a blank `<div>` while data loads (recent change for `/app/domains/[domainId]`). The empty-during-load behavior affects how axe sees the route; we should wait for loaded state before snapshotting.

## Brand color bucket sweep

All 230 occurrences of Dusk `#BBABF4`, Sunset `#DB5835`, and Sunrise `#F29836`, classified by heuristic role. **No fixes proposed in this table** — awaiting per-occurrence review.

### Role legend

- **background/fill** — the hex is set as a `background`, `backgroundColor`, `bg-[...]`, or SVG `fill` value. Decorative for the hex itself; the contrast question is whatever foreground sits on top of it.
- **border** — the hex is used as a `border` or `borderColor` value. WCAG 1.4.11 non-text contrast threshold is 3:1; thin borders also have aesthetic latitude.
- **stroke** — SVG stroke. Same 3:1 threshold as borders.
- **gradient-stop** — used as a stop in a `<linearGradient>` / `<radialGradient>` or in a `linear-gradient(...)` CSS expression. Decorative.
- **helper-return** — the hex is the return value of a helper that maps activity-progress states to colors. The returned value flows into a `background`/`fill` downstream — decorative.
- **shadow** — used as a box-shadow color. Decorative.
- **accent** — used as a form-control `accent-color` / `accentColor`. Small UI affordance.
- **text** — the hex is set as a `color`, `text-[...]`, or `textDecorationColor`. **These rows need contrast review.**
- **unclassified** — heuristic couldn't determine a role from the single line. Needs manual look.

A row can combine roles (e.g., `background/fill+text` typically means one element's background and another's text color sit on the same JSX line).
